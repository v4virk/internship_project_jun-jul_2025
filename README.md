# Real-Time E-Commerce Website Analytics Pipeline

This repository contains a real-time e-commerce event analytics project. The implementation in the `final_project/` folder simulates user activity on an e-commerce website, publishes those events to Kafka, consumes the Kafka stream with Spark Structured Streaming, stores streamed data in HDFS as partitioned Parquet files, and visualizes live analytics through a Streamlit dashboard. A PyQt5 desktop control panel is also implemented to start, stop, monitor, and open the different services used by the pipeline.

The project is implemented as a local big-data pipeline using Python, Kafka, Spark, HDFS, Streamlit, Altair, Pandas, and PyQt5.

---

## What Is Being Implemented

The `final_project/` folder implements a complete real-time event analytics workflow for an e-commerce use case.

The implemented workflow is:

1. A Python event producer continuously generates fake e-commerce events.
2. Every generated event is written as JSON into a local log file.
3. The same event is also sent to a Kafka topic named `test-events`.
4. A Spark Structured Streaming consumer reads JSON messages from Kafka.
5. Spark parses the JSON events using a defined schema.
6. Spark writes the parsed stream to HDFS in Parquet format.
7. The HDFS Parquet output is partitioned by `event_type`.
8. A Streamlit dashboard reads the event log file and displays real-time analytics.
9. A PyQt5 desktop application works as the main control panel for deployment, monitoring, service links, and shutdown.

In short, the project demonstrates how website activity data can move from event generation to Kafka, then into Spark, then into HDFS, and finally into analytics dashboards.

## Event Data Model

The generated events represent user actions on an e-commerce website.

### Event Fields

Each event contains:

| Field | Meaning |
| --- | --- |
| `event_id` | Random user/event identifier such as `U9168`. |
| `event_type` | Type of user action. |
| `timestamp` | Unix timestamp generated using `time.time()`. |
| `products` | Optional list of products related to the event. Login and logout events do not include products. |

### Supported Event Types

The producer generates these event types:

```text
view_product
add_to_cart
purchase
search
login
logout
```

### Product Fields

For events that contain product information, each product can contain:

| Field | Meaning |
| --- | --- |
| `product_id` | Random product identifier such as `P161`. |
| `product_category` | Product category such as Electronics, Fashion, Home, Books, Beauty, Toys, Grocery, Furniture, or Sports and Fitness. |
| `price` | Random price stored as a string, for example `$5523`. |
| `product_type` | Optional field for selected categories. Electronics can have Smartphone, Laptop, or Camera. Fashion can have Shoes, Shirt, or Dress. |
| `brand` | Optional field for selected categories. Electronics can have Apple, Samsung, or Sony. Fashion can have Nike, Zara, or Puma. |

### Sample Event

```json
{
  "event_id": "U9168",
  "event_type": "purchase",
  "timestamp": 1751455834.427258,
  "products": [
    {
      "product_id": "P161",
      "product_category": "Electronics",
      "price": "$5523",
      "product_type": "Smartphone",
      "brand": "Apple"
    },
    {
      "product_id": "P542",
      "product_category": "Grocery",
      "price": "$4746"
    }
  ]
}
```

## How Each Part Works

## 1. Event Producer: `generated_events.py`

`generated_events.py` is responsible for generating random e-commerce activity.

It imports:

- `time` for timestamps and sleep intervals.
- `json` for converting Python dictionaries to JSON.
- `random` for generating random event values.
- `KafkaProducer` from the `kafka-python` package for sending messages to Kafka.
- `atexit` for registering a cleanup/upload function when the script exits.
- `subprocess` and `os` for HDFS upload operations.

The producer is initialized like this:

```python
producer = KafkaProducer(
    bootstrap_servers='localhost:9092',
    value_serializer=lambda v: json.dumps(v).encode('utf-8')
)
```

This means:

- Kafka must be running on `localhost:9092`.
- Every Python event dictionary is serialized into JSON.
- The JSON is encoded as UTF-8 bytes before it is sent to Kafka.

The script continuously runs inside a `while True` loop. On every loop:

1. It randomly chooses an event type from `view_product`, `add_to_cart`, `purchase`, `search`, `login`, and `logout`.
2. It creates an event id in the format `U####`.
3. It adds the current Unix timestamp.
4. If the event is not `login` or `logout`, it adds one to three products.
5. For each product, it randomly assigns product id, category, and price.
6. If the category is Electronics, it can add product type and brand.
7. If the category is Fashion, it can add product type and brand.
8. The event is printed in the terminal.
9. The event is appended to `events_log.json` as one JSON object per line.
10. The event is sent to Kafka topic `test-events`.
11. The script sleeps for a random interval between `0.5` and `5` seconds.

The local log write is implemented as:

```python
with open("events_log.json", "a", encoding="utf-8") as file:
    file.write(json_event + "\n")
    file.flush()
```

The Kafka send is implemented as:

```python
producer.send('test-events', event)
producer.flush()
```

The script also defines an `upload_to_hdfs()` function. This function is registered with `atexit`, so it runs when the script terminates normally. The function attempts to create this HDFS directory:

```text
events_log
```

Then it attempts to upload:

```text
events/events_log.json
```

to:

```text
events_log.json
```

Important note: the script writes runtime logs to `events_log.json`, but the HDFS upload function currently tries to upload `events/events_log.json`. If the `events/` folder does not exist or the file is not present inside it, the upload function will fail. Either create that path or change the upload path to match the actual generated log file.

## 2. Spark Kafka Consumer: `spark_kafka_to_hdfs.py`

`spark_kafka_to_hdfs.py` implements the streaming layer of the project.

It creates a Spark session named `KafkaToHDFSConsumer`:

```python
spark = SparkSession.builder \
    .appName("KafkaToHDFSConsumer") \
    .config("spark.jars.packages", "org.apache.spark:spark-sql-kafka-0-10_2.12:3.3.2,org.apache.kafka:kafka-clients:3.3.2") \
    .config("spark.hadoop.fs.defaultFS", "hdfs://localhost:9000") \
    .getOrCreate()
```

This Spark session is configured to:

- Use the Spark Kafka connector package.
- Use Kafka client libraries.
- Use HDFS at `hdfs://localhost:9000` as the default filesystem.

The script defines a schema for the incoming Kafka JSON messages. The schema includes:

- `event_id` as string.
- `event_type` as string.
- `timestamp` as double.
- `products` as an optional array of product structs.

The Spark stream reads from Kafka using:

```python
df = spark.readStream \
    .format("kafka") \
    .option("kafka.bootstrap.servers", "localhost:9092") \
    .option("subscribe", "test-events") \
    .option("startingOffsets", "earliest") \
    .load()
```

This means:

- Kafka must be available on `localhost:9092`.
- Spark subscribes to the topic `test-events`.
- Spark starts reading from the earliest available offsets.

Kafka stores the message payload in a binary column named `value`, so the script converts `value` into a string and parses it as JSON:

```python
parsed = df.selectExpr("CAST(value AS STRING)") \
    .select(from_json(col("value"), schema).alias("data")) \
    .select("data.*")
```

The parsed stream is then written to HDFS:

```python
query = parsed.writeStream \
    .format("parquet") \
    .option("path", "events/") \
    .option("checkpointLocation", "checkpoints/") \
    .partitionBy("event_type") \
    .outputMode("append") \
    .start()
```

This means:

- Output format is Parquet.
- Events are stored in the HDFS output folder named `events/`.
- Spark checkpoint data is stored in the checkpoint folder named `checkpoints/`.
- Data is partitioned by `event_type`.
- New records are appended as they arrive.

The partitioned HDFS output will look conceptually like:

```text
events/
├── event_type=add_to_cart/
├── event_type=login/
├── event_type=logout/
├── event_type=purchase/
├── event_type=search/
└── event_type=view_product/
```

The query runs continuously using:

```python
query.awaitTermination()
```

## 3. Streamlit Analytics Dashboard: `app.py`

`app.py` implements a browser-based real-time analytics dashboard using Streamlit, Pandas, and Altair.

The dashboard reads event logs from this hardcoded file path:

```text
events_log.json
```

The file is loaded by the cached function `load_events()`:

```python
@st.cache_data(ttl=5)
def load_events():
    events = []
    with open(EVENTS_FILE, 'r') as f:
        for line in f:
            event = json.loads(line.strip())
            event['timestamp'] = datetime.fromtimestamp(event['timestamp'])
            event['product_categories'] = [p.get('product_category', 'Unknown') for p in event.get('products', [])]
            events.append(event)
    return pd.DataFrame(events)
```

This function:

1. Opens the event log file.
2. Reads each line as one JSON event.
3. Converts the Unix timestamp into a Python `datetime` value.
4. Extracts product categories from the nested `products` list.
5. Returns all events as a Pandas DataFrame.
6. Caches the result for five seconds.

The dashboard page is configured as:

```python
st.set_page_config(page_title="Real-Time Event Dashboard", layout="wide")
```

The dashboard title is:

```text
📊 Real-Time Event Analytics Dashboard
```

### Dashboard Controls

The sidebar contains:

- Analysis time window selector.
- Auto-refresh checkbox.
- Refresh rate slider.
- Data status section.
- Event type multi-select filter.
- Option to use only the last 60 seconds of data.

Supported analysis windows are:

```text
30min
1h
4h
12h
24h
7d
All
```

If a time window is selected, the dashboard filters rows where `timestamp` is greater than the selected threshold.

### Dashboard Metrics

The dashboard displays three summary metrics:

1. Total Events.
2. Unique Event Types.
3. Total Products Involved.

### Dashboard Charts

The dashboard implements these visualizations:

#### Event Type Distribution

A horizontal Altair bar chart showing the number of events for each event type.

#### Product Category Distribution

A horizontal Altair bar chart showing how many times each product category appears in the filtered events.

#### Events per Second

A time-series line chart created by resampling events at one-second frequency.

#### Top Product Categories Over Time

A line chart for the top five product categories over time, grouped in five-second intervals.

#### User Journey Funnel

The dashboard categorizes user behavior into:

- `Completed Purchase`: events contain both `login` and `purchase`.
- `Abandoned Cart`: events contain both `login` and `add_to_cart`.
- `Browsing Only`: events contain `login` only.
- `Other`: all other paths.

Important note: the current implementation groups journeys by `event_id`. Since the producer generates random `event_id` values for each event, this is an approximate journey calculation rather than a true long-term user session calculation.

#### Top 5 Active Users

A bar chart showing the top five most frequent `event_id` values.

#### Category vs Event Type Heatmap

A heatmap showing counts between product categories and event types.

#### Event Types Over Time

A stacked area chart showing event type counts over time using five-second intervals.

### Auto Refresh

If auto-refresh is enabled, the dashboard waits for the selected refresh interval and reruns itself:

```python
if auto_refresh:
    time.sleep(refresh_rate)
    st.rerun()
```

## 3A. Parquet-Based Streamlit Dashboard: `parquet_streamlit_app.py`

`parquet_streamlit_app.py` is the visualization-ready program for reading the Parquet files generated by the Spark streaming pipeline.

This dashboard does not read the raw JSON log file. Instead, it reads the Spark Parquet output folder.

The purpose of this file is to connect the visualization layer directly with the processed data written by Spark.

### How It Reads Parquet Data

The app creates a reusable Spark session inside Streamlit:

```python
SparkSession.builder.appName("ParquetEventAnalyticsDashboard").getOrCreate()
```

Then it reads the Parquet folder using:

```python
spark.read.parquet(parquet_path)
```

The `parquet_path` is entered from the Streamlit sidebar. By default, it is:

```text
events/
```

This path should point to the folder where `spark_kafka_to_hdfs.py` writes Parquet files.

If Spark writes partitioned folders such as:

```text
events/
├── event_type=add_to_cart/
├── event_type=login/
├── event_type=purchase/
└── event_type=view_product/
```

the Parquet dashboard reads the full `events/` folder and Spark automatically understands the partitioned data.

### How It Makes Data Visualization-Ready

After reading the Parquet data, the app converts the Spark DataFrame into a Pandas DataFrame:

```python
spark_df.toPandas()
```

Then it prepares the data for charts by:

1. Ensuring `event_type` exists.
2. Ensuring `event_id` exists.
3. Converting `timestamp` into readable datetime format.
4. Normalizing the nested `products` column.
5. Extracting product categories into a chart-friendly `product_categories` column.
6. Creating a `product_count` column.
7. Filtering data by time window and selected event types.

The nested product data is converted into normal Python dictionaries so product category charts, heatmaps, and product metrics can be created easily.

### Visualizations Provided

The Parquet dashboard provides:

- Total events metric.
- Unique event types metric.
- Total products metric.
- Unique event IDs metric.
- Prepared data preview table.
- CSV download for prepared data.
- Event type distribution chart.
- Product category distribution chart.
- Events over time chart.
- Top product categories over time chart.
- User journey funnel chart.
- Top active event IDs chart.
- Category vs event type heatmap.
- Event types over time area chart.

### Auto Refresh

The app supports auto-refresh. When enabled, it waits for the selected refresh interval and reruns:

```python
time.sleep(refresh_rate)
st.rerun()
```

This allows the dashboard to reload newly written Parquet files while the Spark streaming job is running.

## How to Run the Project Manually

These steps run the pipeline without using the PyQt5 frontend.

### 1. Start Kafka

Start Kafka locally. One command used in the frontend is:

```bash
brew services start kafka
```

If your Kafka setup requires manual Zookeeper startup, start Zookeeper as well.

### 2. Start HDFS

```bash
start-dfs.sh
```

### 3. Start YARN

```bash
start-yarn.sh
```

### 4. Create or Verify the Kafka Topic

The project sends events to topic `test-events`. Create it if it does not already exist:

```bash
kafka-topics.sh --create --topic test-events --bootstrap-server localhost:9092 --partitions 1 --replication-factor 1
```

### 5. Start the Event Producer

From the repository root, run:

```bash
python final_project/generated_events.py
```

This starts generating events, writing them to `events_log.json`, and sending them to Kafka.

### 6. Start the Spark Streaming Consumer

In another terminal, run:

```bash
spark-submit \
  --packages org.apache.spark:spark-sql-kafka-0-10_2.12:3.3.2,org.apache.kafka:kafka-clients:3.3.2 \
  final_project/spark_kafka_to_hdfs.py
```

This starts reading events from Kafka and writing Parquet data into HDFS.

### 7. Start the JSON-Based Streamlit Dashboard

In another terminal, run:

```bash
streamlit run final_project/app.py
```

Then open:

```text
http://localhost:8501
```

Important note: before running Streamlit, make sure `EVENTS_FILE` inside `final_project/app.py` points to the same event log file that the producer is writing.

### 7A. Start the Parquet-Based Streamlit Dashboard

To visualize the Parquet files produced by Spark, run:

```bash
streamlit run final_project/parquet_streamlit_app.py
```

In the Streamlit sidebar, set the Parquet folder path to the folder where Spark writes the Parquet output. By default, the app uses:

```text
events/
```

This dashboard reads the Parquet folder, converts the Spark data into a visualization-ready Pandas DataFrame, extracts nested product categories, and creates charts from the processed Parquet data.

### 8. Read HDFS JSON Logs with Spark

To run the batch reader:

```bash
python final_project/read_event_logs.py
```

This reads the JSON log path configured in `read_event_logs.py` and prints the DataFrame.

## Current Limitations and Notes

1. Several paths are hardcoded in the scripts and should be changed to configurable file names before sharing or deploying.
2. `frontend.py` should point to the project files `generated_events.py`, `spark_kafka_to_hdfs.py`, and `app.py`.
3. `app.py` reads `events_log.json`, and `generated_events.py` should write to the same `events_log.json` file for the dashboard to show live data.
4. `generated_events.py` writes `events_log.json`, but its HDFS upload function tries to upload `events/events_log.json`.
5. The file monitor in `frontend.py` attempts to parse the whole event log file as one JSON object, but the event log is actually newline-delimited JSON.
6. The journey funnel groups by `event_id`, but the producer creates random event ids, so this is not a complete user-session model.
7. `read_event_logs.py` expects fields such as `ip`, `device`, and `user_type`, but the current event producer does not generate those fields.
8. The `configuration` file currently exists but does not contain configuration values.

## Summary

This project implements a real-time analytics system for simulated e-commerce website events. It generates events with Python, streams them through Kafka, processes them with Spark Structured Streaming, stores them in HDFS as partitioned Parquet files, and visualizes activity using Streamlit. A PyQt5 desktop frontend is included to control the local services, start and stop pipeline components, monitor logs, and open service dashboards.
